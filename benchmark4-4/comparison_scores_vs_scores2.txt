WHAT STILL REMAINS
--------------------------------------------------------------------------------
- fruitbowl output=>original still 0/35 + 0/20. The field-placement mismatch is
  only resolved in one orientation: when the instance (from the output) declares
  color/quality/size on the parent Fruit but the reference model declares them on
  the children apple/orange, the "field of Sig" containment check still rejects it.

- marksweepgc still 9/119, all instances 0. Root cause is now DIFFERENT from the
  old subset collision: it is the cross-model extends-vs-`in` mismatch. The
  reference has `one sig root extends Node` (primary) while the output has
  `one sig root in Node` (subset). Reconstructing one model's instance emits
  `one sig <atom> extends root`, which is illegal against the other model where
  `root` is a subset sig:
      FAIL: Cannot extend the subset signature "this/root".
  (This is the same extends-vs-`in` class that scripts/extends_to_in.py normalizes
  for Ringert, but the InstanceChecker does not apply that normalization.)

- FileSystem unchanged at 45/115: same cross-model extends-vs-`in` situation on
  `Alias` (ref `Alias extends File` vs output `Alias in File`); instance checks
  stay partial. Ringert is 5/5 both ways (from the extends_to_in fix).

- Genuine, non-artifact losses are untouched (as expected): COFFEE out=>orig
  Ringert 0/4 (too permissive), ex_Arity out=>orig Ringert 0/5 + 900s timeout
  (dropped total-function fact), alloy4fun orig=>out over-restriction.

BOTTOM LINE
--------------------------------------------------------------------------------
The update resolved the non-empty-subset reconstruction crash, giving big gains
(ThreeOfAKind perfect, alloy4fun +77, fruitbowl +55). Two harness gaps remain:
(1) field placement is only matched in one direction across an extends hierarchy
(fruitbowl), and (2) the checker still cannot compare instances when the two
models disagree on extends vs `in` for the same sig (marksweepgc, FileSystem) -
it needs the same extends->in normalization already used for Ringert.
